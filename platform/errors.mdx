## Full error code reference

| Status | `error.code` | Common causes | Fix |
| ------ | ------------ | ------------- | --- |
| `400` | `invalid_request` | Missing required parameter, malformed JSON body, invalid date format | Check the `error.message` for which parameter is invalid. Refer to the endpoint's API reference for required fields. |
| `401` | `unauthorized` | Missing `Authorization` header, expired API key, malformed bearer token | Ensure the header is `Authorization: Bearer <key>`. Regenerate your key at [irora.ai/keys](https://irora.ai/keys) if expired. |
| `403` | `forbidden` | API key scoped to a different product, insufficient plan tier | Verify your key matches the product you're calling. Upgrade your plan if the endpoint requires a higher tier. |
| `404` | `not_found` | Invalid filing ID, deleted watchlist, typo in endpoint path | Double-check the resource ID and endpoint URL. Use the search endpoint to find valid IDs. |
| `409` | `conflict` | Duplicate watchlist name, alert already exists for this filing | Change the conflicting field value or delete the existing resource first. |
| `422` | `unprocessable` | Valid JSON but semantically wrong (e.g., `limit: -1`, future date for historical endpoint) | Fix the field value per the `error.message` guidance. |
| `429` | `rate_limited` | Exceeded per-minute or per-day request quota | Check `X-RateLimit-Remaining` and `X-RateLimit-Reset` headers. Implement exponential backoff. See [rate limits](/platform/rate-limits). |
| `500` | `internal_error` | Unexpected server failure | Retry after a few seconds. If persistent, contact support with the `meta.requestId`. |
| `502` | `upstream_error` | SEC EDGAR or upstream data provider is unavailable | Cached data may still be returned (check `X-Data-Source: cache` header). Monitor [status.irora.ai](https://status.irora.ai). |
| `503` | `service_unavailable` | Planned maintenance or deploy in progress | Retry in 30–60 seconds. Check [status.irora.ai](https://status.irora.ai) for maintenance windows. |

## Debugging errors

### Use the request ID

Every response includes a `meta.requestId` field. Include this in support tickets for faster resolution.

```bash
curl -s https://api.irora.ai/irora-sec/v1/health | jq '.meta.requestId'
# "req_abc123def456"
```

### Common mistakes

<AccordionGroup>
  <Accordion title="401 Unauthorized — missing API key">
    Ensure the `Authorization` header is set with `Bearer ` prefix:
    ```bash
    # Wrong
    curl -H "Authorization: my_key" ...

    # Correct
    curl -H "Authorization: Bearer my_key" ...
    ```
  </Accordion>
  <Accordion title="403 Forbidden — wrong product key">
    API keys are scoped per product. Ensure you are using the correct key for the product you are accessing.
  </Accordion>
  <Accordion title="429 Too Many Requests — rate limited">
    Check the `X-RateLimit-Remaining` header. If zero, wait until `X-RateLimit-Reset` before retrying. See [rate limits](/platform/rate-limits) for per-tier limits.
  </Accordion>
  <Accordion title="502 Upstream Error — EDGAR is down">
    Some endpoints rely on upstream SEC EDGAR feeds. When EDGAR is unavailable, cached data is returned with a `X-Data-Source: cache` header. Check [status.irora.ai](https://status.irora.ai) for real-time status.
  </Accordion>
</AccordionGroup>
